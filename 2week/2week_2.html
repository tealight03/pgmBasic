• *Jenkins*

    프로젝트 빌드 작업을 자동화해주는 툴이다. 빌드는 작업하는 데 꽤 많은 시간을 소요하므로
    개발자들은 젠킨스를 통해 빌드를 자동화하고, 파이프라인(Pipeline)을 사용해 거의 모든 언어의 조합과
    소스코드 리포지토리에 대한 지속적인 통합과 지속적인 전달 환경을 구축할 수 있게 된다.
    젠킨스는 개발자들에게 여러가지 배포 작업의 단계별 상황을 모니터링할 수 있는 대쉬보드,
    배포 스크립트를 실행해주는 기능, 다양한 개발환경을 위한 플러그인 들을 제공한다.
    다만 젠킨스를 구축하는 것이 조금 까다롭다고 한다.
    AWS의 EC2에서 젠킨스 서버를 구축해서 운영하는 방식이라는데...
    지금은 그나마 쉬워졌지만 그렇다고 해서 초보자가 환경 설정하기 쉬운 접근 구조는 아니라고 한다.

  • tmux 환경

    tmux는 "terminal Multiplexer"의 줄임말로 터미널에서 개발을 진행 할 때 유용한 기능을 제공하는 응용 커맨드 소프트웨어이다.
    screen 명령어와 기능이 매우 유사하여 한 터미널에서 session을 실행 시키면 그 세션 안에서 작업 창(window)을 여러개 나누어서 병렬적으로 수행할 수 있다.
    이 윈도우에서 또 pane이라는 화면 분할도 가능하고, 이렇게 나누어서 작업하는 것이 tmux의 핵심 키워드라고 할 수 있다.
    그렇게 되면 terminal 어플리케이션을 여러 개 실행시키는 것보다 프로세스 메모리 측면에서 큰 효율을 얻을 수 있으며 단순히 창을 나누는 기능 뿐만 아니라,
    서버에서 시간이 오래 걸리는 작업을 백그라운드로 켜놓고 터미널 종료 후 다른 작업을 할 수 있다는 장점도 있기 때문에 개발자라면 많이 사용하는 툴이다.

  • git과 SVN

    git과 SVN은 둘 다 버전 관리 시스템인데, git은 분산형 버전 관리 시스템이고 SVN은 중앙 집중식 버전 관리 시스템이다.
    git이 SVN보다 좀 더 최신 버전 관리 시스템이다. 분산형이기 때문에 개발자들이 언제 어디서나 작업을 할 수 있고, SVN보다 빠르고 SVN보다 다양한 기능을 제공한다.
    SVN은 Git보다 오래된 버전 관리 시스템이고, 사용하기 쉽고, Git보다 많은 사용자를 보유하고 있다.
    git과 SVN을 비교하자면 git은 SVN보다 유연하고 강력하지만, 사용자에 따라 사용하기 어려울 수도 있고,
    SVN은 git보다 사용하기 쉽지만, git보다 유연하고 강력하지 않습니다. 둘 다 각각의 장단점을 모두 갖고 있기 때문에
    프로젝트가 규모가 크고, 유연하고 강력한 버전 관리 시스템이 필요한 경우 Git을 사용하는 것이 좋지만
    프로젝트가 규모가 작고, 사용하기 쉬운 버전 관리 시스템이 필요한 경우 SVN을 사용하는 것이 좋다.

  • rebase

    깃허브에서 브랜치끼리 버전을 병합하는 방법에는 merge와 rebase가 있다.
    두 명령어는 브랜치의 마지막 버전끼리 병합하는 명령어라는 공통점이 있지만,
    병합 이후 결과물(?)이 조금 다르게 나타난다. merge는 그전에 했던 commit들을 모두 남겨놓지만,
    rebase는 병합 이후 필요 없는 commit들을 모두 삭제하기 때문에 좀 더 깔끔하게 git을 관리할 수 있다.
    그렇지만 프로젝트용으로 쓰는 브랜치에 git rebase를 하면 내가 rebase한 내용을 다른 사람들이 git pull로
    당겨받았을 때 conflict를 일으킬 수 있기 때문에 내 local에서 작업하고 origin(main)에 push하기 전
    커밋 정리 용도 정도로 rebase를 사용하는 것이 좋다.

  • Gradle과 Maven

    Gradle과 Maven은 프로젝트 빌드 및 종속성 관리 도구로, Java 기반 프로젝트의 개발 및 빌드 프로세스를 자동화할 때 사용된다.
    프로젝트의 라이프사이클, 의존성 관리 및 테스트 실행, 배포 등의 작업을 수행하는 기능을 제공한다.
    먼저 Maven은 아파치 소프트웨어 재단에서 개발하는 Java 기반 프로젝트의 라이프사이클 관리를 위한 빌드 도구로서
    컴파일과 빌드를 동시에 수행, 혹은 테스트를 병행하거나 서버 측 Deploy 자원을 관리할 수 있는 환경을 제공한다.
    Java로 개발하다 보면 다양한 라이브러리를 필요로 하게 되는데, Maven은 라이브러리 관리 기능도 내포하고 있어서
    settings.xml 또는 pom.xml 파일에 필요한 라이브러리만 적으면 Maven이 알아서 다운받고 설치해주고 경로도 지정해준다.
    Gradle은 안드로이드 스튜디오의 기본 빌드 툴로, 빌드 스크립트의 유연성, 빌드 속도 및 성능, 다양한 플러그인 생태계,
    스크립트 기반으로 프로젝트를 구성한다는 점에서 Maven의 단점을 보완한다. Gradle을 이용하면 더욱 효율적인 빌드 프로세스를 사용할 수 있다.

  • Azure

    Microsoft Azure는 마이크로소프트사가 제공하는 클라우드 컴퓨팅 플랫폼, 서비스를 지칭한다.
    2011년 PaaS에 이어 2013년 IaaS 서비스를 시작했고, 아마존 웹 서비스(AWS) 등과 경쟁하고 있다.
    인프라, 플랫폼, 소프트웨어 서비스를 제공하여 개발자가 애플리케이션을 빌드, 배포 및 관리할 수 있도록 지원하고
    그 외에도 다양한 서비스와 기능을 제공해 개발자가 필요한 리소스를 확장하고, 데이터를 저장하고, 안전하게 실행할 수 있도록 한다.

  • AWS Elastic

    AWS Elastic은 Amazon Web Services에서 제공하는 다양한 컴퓨팅, 스토리지, 네트워킹, 데이터베이스, 분석, 머신 러닝, 인공 지능,
    보안, 관리, 네트워킹, 개발, 빌드, 테스트, 배포, 모니터링, 비즈니스 인텔리전스, 기타 서비스를 제공하고, 비즈니스 규모에 관계없이
    모든 규모의 기업과 조직이 클라우드에서 애플리케이션을 개발, 배포 및 관리할 수 있도록 해준다.

  • JUnit

    자바 언어용으로 개발된 단위 테스트 프레임워크이다. 참고로 단위 테스트 프레임 워크란,
    각각의 함수, 메서드, 클래스 또는 모듈과 같은 작은 코드 단위에 대해 테스트 케이스를 작성하고 실행하여 코드의 동작을 검증할 때 사용되는 프레임워크를 말한다.
    주닛(JUnit)은 테스트 주도 개발(Test-Driven Development, TDD) 및 애자일 소프트웨어 개발 방법론에서 주로 사용되며,
    자바 코드의 품질을 향상시키고 버그를 찾아내기 위해 사용된다. 주닛의 애노테이션('@')을 이용한 테스트 케이스 작성,
    테스트 그룹화, 테스트 커버리지 측정 등을 이용해 코드를 검증하고 유지 관리할 수 있다.
    일반적으로 각 언어에는 그 언어만을 위한 단위 테스트 프레임워크가 존재한다고 한다.

  • TestNG

    Java 언어를 위한 유연하고 강력한 단위 테스트 프레임워크입니다.
    "NG"는 "Next Generation"을 의미하며, JUnit 프레임워크의 확장된 기능과 대안을 제공한다.
    TestNG는 Java의 테스트 코드 작성 및 실행, 그리고 테스트 결과 분석을 지원한다.
    주로 대규모 프로젝트에서 사용되며, 테스트 수명주기 관리, 병렬 실행, 테스트 그룹화 및 필터링 등의 기능을 이용해
    좀 더 강력한 단위 테스트를 작성하고 실행할 수 있고, CI/CD 환경에서의 테스트 자동화에도 사용된다.

  • GitLab

    국내에선 다소 인지도가 낮은 무료 Git 저장소. Ruby on Rails 로 만들어진 오픈소스 저장소 솔루션이다.
    참여 인원 수에 관계없이 비공개 저장소를 무제한으로 생성할 수 있어 이곳을 이용하는 기업들이 많이 늘었다.
    Bitbucket은 저장소의 최대 용량가 2GB인 반면 GitLab은 5GB이고(무료 사용자 기준), 또한 GitHub보다 훨씬 체계적인 이슈 트래커 기능과 CI(지속적 통합) 서비스 등을 제공한다.
    GitLab CE, GitLab EE라는 설치형 소프트웨어가 있으며, GitLab CE는 무료로, EE는 유료로 사용할 수 있다. 국내 세금계산서가 필요한 경우 국내 총판에서 상담하면 된다.

  • AWS ELB

    AWS ELB("Elastic Load Balancing")은 AWS에서 제공하는 로드 밸런싱 서비스로,
    여러 개의 백엔드 서버에 들어오는 트래픽을 분산하여 각 서버에 균일한 부하를 분산해주는 역할을 한다.
    ELB는 로드 밸런싱을 위해 Application Load Balancer, Network Load Balancer, Classic Load Balancer의 3가지 유형으로 제공된다.
    또한 백엔드 서버의 상태를 모니터링하여 장애가 발생한 서버를 자동으로 감지하고 트래픽을 다른 서버로 전환하기도 한다.
    ELB를 사용하면 ELB를 AWS EC2 인스턴스와 함께 사용하면 웹 애플리케이션을 실행할 수도 있고,
    AWS RDS 인스턴스와 함께 사용하면 데이터베이스 서버를 실행하는 등 애플리케이션의 성능과 가용성을 향상시키고, 애플리케이션의 관리를 간소화할 수 있다.
